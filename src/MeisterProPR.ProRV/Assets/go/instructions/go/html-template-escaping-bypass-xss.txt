Review focus: Cross-site scripting via HTML template escaping bypass

What to look for:
- Converting user input to a special type that avoids escaping when fed into an HTML template allows for a cross-site scripting vulnerability.
- In Go, the`html/template`package has a few special types (`HTML`,`HTMLAttr`,`JS`,`JSStr`,`CSS`,`Srcset`, and`URL`) that allow values to be rendered as-is in the template, avoiding the escaping that all the other strings go through. Using them on user-provided values allows for a cross-site scripting vulnerability.
- Keep the check only when the changed code appears to introduce, remove, bypass, or misuse the risky behavior itself.
- Prefer explicit code evidence such as API calls, removed guards, changed data flow steps, changed arguments, or changed signatures over broad similarity.

Relevance threshold:
- Drop this check if the diff only changes naming, comments, formatting, or unrelated plumbing.
- Do not keep this check solely because the file belongs to a broad category like security, configuration, or API usage.
- This is path-sensitive. Keep it only when the diff suggests a plausible source-to-sink or input-to-operation path worth deeper inspection.

Reference examples:
- Reference example (`HtmlTemplateEscapingBypassXssBad.go`):
```
package main

import (
	"html/template"
	"net/http"
)

func bad(w http.ResponseWriter, r *http.Request) {
	r.ParseForm()
	username := r.Form.Get("username")
	tmpl, _ := template.New("test").Parse(`<b>Hi {{.}}</b>`)
	tmpl.Execute(w, template.HTML(username))
}
```
- Reference example (`HtmlTemplateEscapingBypassXssGood.go`):
```
package main

import (
	"html/template"
	"net/http"
)

func good(w http.ResponseWriter, r *http.Request) {
	r.ParseForm()
	username := r.Form.Get("username")
	tmpl, _ := template.New("test").Parse(`<b>Hi {{.}}</b>`)
	tmpl.Execute(w, username)
}
```
