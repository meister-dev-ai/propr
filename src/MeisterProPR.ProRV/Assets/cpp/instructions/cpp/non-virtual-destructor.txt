Review focus: Non-virtual destructor

What to look for:
- When a class and its derived class both define non-virtual destructors, the destructor of the derived class may not always be called.
- This rule finds classes that define a non-virtual destructor, yet they have derived classes that also define a non-virtual destructor. This can prevent proper cleanup of resources as only the destructor of the type of the variable will be called (instead of the type of the object instance).
- Keep the check only when the changed code appears to introduce, remove, bypass, or misuse the risky behavior itself.
- Prefer explicit code evidence such as API calls, removed guards, changed data flow steps, changed arguments, or changed signatures over broad similarity.

Relevance threshold:
- Drop this check if the diff only changes naming, comments, formatting, or unrelated plumbing.
- Do not keep this check solely because the file belongs to a broad category like security, configuration, or API usage.
- This is direct-pattern focused. Keep it only when the diff shows the unsafe call, broken contract, or misuse pattern itself.

Reference examples:
- Reference example (`NonVirtualDestructor.cpp`):
```
class Base {
public:
    Resource *p;
    Base() {
        p = createResource();
    }
    //...
    ~Base() {
        //wrong: this destructor is non-virtual, but Base has a derived class
        //with a non-virtual destructor
        freeResource(p);
    }
};

class Derived: public Base {
public:
    Resource *dp;
    Derived() {
        dp = createResource2();
    }
    ~Derived() {
        freeResource2(dp);
    }
};

int f() {
    Base *b = new Derived(); //creates resources for both Base::p and Derived::dp
    //...
    delete b; //will only call Base::~Base(), leaking the resource dp.
        // Change both destructors to virtual to ensure they are both called.
}
```
