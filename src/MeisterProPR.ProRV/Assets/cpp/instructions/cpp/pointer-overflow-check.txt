Review focus: Pointer overflow check

What to look for:
- Adding a value to a pointer to check if it overflows relies on undefined behavior and may lead to memory corruption.
- When checking for integer overflow, you may often write tests like`p + i < p`. This works fine if`p`and`i`are unsigned integers, since any overflow in the addition will cause the value to simply "wrap around." However, using this pattern when`p`is a pointer is problematic because pointer overflow has undefined behavior according to the C and C++ standards. If the addition overflows and has an undefined result, the comparison will likewise be undefined; it may produce an unintended result, or may be deleted entirely by an optimizing compiler.
- Keep the check only when the changed code appears to introduce, remove, bypass, or misuse the risky behavior itself.
- Prefer explicit code evidence such as API calls, removed guards, changed data flow steps, changed arguments, or changed signatures over broad similarity.

Relevance threshold:
- Drop this check if the diff only changes naming, comments, formatting, or unrelated plumbing.
- Do not keep this check solely because the file belongs to a broad category like security, configuration, or API usage.
- This is direct-pattern focused. Keep it only when the diff shows the unsafe call, broken contract, or misuse pattern itself.

Reference examples:
- Bad example (`PointerOverflow-bad.cpp`):
```
bool not_in_range(T *ptr, T *ptr_end, size_t i) {
    return ptr + i >= ptr_end || ptr + i < ptr; // BAD
}
```
- Good example (`PointerOverflow-good.cpp`):
```
bool not_in_range(T *ptr, T *ptr_end, size_t i) {
    return i >= ptr_end - ptr; // GOOD
}
```
