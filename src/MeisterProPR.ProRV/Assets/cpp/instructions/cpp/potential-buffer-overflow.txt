Review focus: Potential buffer overflow

What to look for:
- Using a library function that does not check buffer bounds requires the surrounding program to be very carefully written to avoid buffer overflows.
- This rule highlights potentially overflowing calls to the functions`sprintf`and`vsprintf`with a warning. These functions allow unbounded writes to buffers, which may cause an overflow when used on untrusted data or without adequate checks on the size of the data. Function calls of this type constitute a security risk through buffer overflows.
- Keep the check only when the changed code appears to introduce, remove, bypass, or misuse the risky behavior itself.
- Prefer explicit code evidence such as API calls, removed guards, changed data flow steps, changed arguments, or changed signatures over broad similarity.

Relevance threshold:
- Drop this check if the diff only changes naming, comments, formatting, or unrelated plumbing.
- Do not keep this check solely because the file belongs to a broad category like security, configuration, or API usage.
- This is direct-pattern focused. Keep it only when the diff shows the unsafe call, broken contract, or misuse pattern itself.

Reference examples:
- Reference example (`PotentialBufferOverflow.cpp`):
```
void f(char* s, float f) {
	char buf[30];

	//wrong: gets has no limit to the length of data it puts in the buffer
	gets(buf); 

	//wrong: sprintf does not limit the length of the string put into buf
	sprintf(buf, "This is a string: %s", s);

	//wrong: %f can expand to a very long string in extreme cases, easily overrunning this buffer
	sprintf(buf, "This is a float: %f", f);
}
```
