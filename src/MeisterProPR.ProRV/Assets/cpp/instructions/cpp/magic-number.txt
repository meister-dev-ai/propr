Review focus: Magic numbers

What to look for:
- 'Magic constants' should be avoided: if a nontrivial constant is used repeatedly, it should be encapsulated into a const variable or macro definition.
- A magic number is a numeric literal (for example,`8080`,`2048`) that is used in the middle of a block of code without explanation. It is considered good practice to avoid magic numbers by assigning the numbers to named constants and using the named constants instead. The reasons for this are twofold:
- A number in isolation can be inexplicable to later programmers, whereas a named constant (such as`MAX_GUESTS`) is more readily understood.
- Using the same named constant in many places, makes the code much easier to update if the requirements change (for example, one more guest is permitted). This rule finds magic numbers for which there is no pre-existing named constant (for example, the line marked (4) below).
- Keep the check only when the changed code appears to introduce, remove, bypass, or misuse the risky behavior itself.
- Prefer explicit code evidence such as API calls, removed guards, changed data flow steps, changed arguments, or changed signatures over broad similarity.

Relevance threshold:
- Drop this check if the diff only changes naming, comments, formatting, or unrelated plumbing.
- Do not keep this check solely because the file belongs to a broad category like security, configuration, or API usage.
- This is direct-pattern focused. Keep it only when the diff shows the unsafe call, broken contract, or misuse pattern itself.

Reference examples:
- Reference example (`MagicConstantsNumbers.cpp`):
```
void sanitize(Fields[] record) {
    //The number of fields here can be put in a const
    for (fieldCtr = 0; field < 7; field++) {
        sanitize(fields[fieldCtr]);
    }
}

#define NUM_FIELDS 7

void process(Fields[] record) {
    //This avoids using a magic constant by using the macro instead
    for (fieldCtr = 0; field < NUM_FIELDS; field++) {
        process(fields[fieldCtr]);
    }
}
```
