Review focus: Signed overflow check

What to look for:
- Testing for overflow by adding a value to a variable to see if it "wraps around" works only for unsigned integer values.
- When checking for integer overflow, you may often write tests like`a + b signed integers is problematic because signed overflow has undefined behavior according to the C and C++ standards. If the addition overflows and has an undefined result, the comparison will likewise be undefined; it may produce an unintended result, or may be deleted entirely by an optimizing compiler.
- Keep the check only when the changed code appears to introduce, remove, bypass, or misuse the risky behavior itself.
- Prefer explicit code evidence such as API calls, removed guards, changed data flow steps, changed arguments, or changed signatures over broad similarity.

Relevance threshold:
- Drop this check if the diff only changes naming, comments, formatting, or unrelated plumbing.
- Do not keep this check solely because the file belongs to a broad category like security, configuration, or API usage.
- This is direct-pattern focused. Keep it only when the diff shows the unsafe call, broken contract, or misuse pattern itself.

Reference examples:
- Bad example (`SignedOverflowCheck-bad1.cpp`):
```
bool foo(int n1, unsigned short delta) {
    return n1 + delta < n1; // BAD
}
```
- Good example (`SignedOverflowCheck-good1.cpp`):
```
#include <limits.h>
bool foo(int n1, unsigned short delta) {
    return n1 > INT_MAX - delta; // GOOD
}
```
- Bad example (`SignedOverflowCheck-bad2.cpp`):
```
bool bar(unsigned short n1, unsigned short delta) {
    // NB: Comparison is always false
    return n1 + delta < n1; // GOOD (but misleading)
}
```
- Good example (`SignedOverflowCheck-good2.cpp`):
```
bool bar(unsigned short n1, unsigned short delta) {
    return (unsigned short)(n1 + delta) < n1; // GOOD
}
```
