Review focus: Magic strings

What to look for:
- 'Magic constants' should be avoided: if a nontrivial constant is used repeatedly, it should be encapsulated into a const variable or macro definition.
- A magic string is a string literal (for example,`"SELECT"`,`"127.0.0.1"`) that is used in the middle of a block of code without explanation. It is considered good practice to avoid magic strings by assigning the strings to named constants and using the named constants instead. The reasons for this are twofold:
- A string in isolation can be inexplicable to later programmers, whereas a named constant (such as`SMTP_HELO`) is more readily understood.
- Using the same named constant in many places, makes the code much easier to update if the requirements change (for example, a protocol is updated). This rule finds magic strings for which there is no pre-existing named constant.
- Keep the check only when the changed code appears to introduce, remove, bypass, or misuse the risky behavior itself.
- Prefer explicit code evidence such as API calls, removed guards, changed data flow steps, changed arguments, or changed signatures over broad similarity.

Relevance threshold:
- Drop this check if the diff only changes naming, comments, formatting, or unrelated plumbing.
- Do not keep this check solely because the file belongs to a broad category like security, configuration, or API usage.
- This is direct-pattern focused. Keep it only when the diff shows the unsafe call, broken contract, or misuse pattern itself.
