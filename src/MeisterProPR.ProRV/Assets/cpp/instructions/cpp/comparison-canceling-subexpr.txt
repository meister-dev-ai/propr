Review focus: Comparison with canceling sub-expression

What to look for:
- If the same sub-expression is added to both sides of a comparison, and there is no possibility of overflow or rounding, then the sub-expression is redundant and could be removed.
- If a comparison contains multiple linear sub-expressions which cancel each other out, then the comparison can be simplified by removing them. This may be indicative of a coding error.
- Keep the check only when the changed code appears to introduce, remove, bypass, or misuse the risky behavior itself.
- Prefer explicit code evidence such as API calls, removed guards, changed data flow steps, changed arguments, or changed signatures over broad similarity.

Relevance threshold:
- Drop this check if the diff only changes naming, comments, formatting, or unrelated plumbing.
- Do not keep this check solely because the file belongs to a broad category like security, configuration, or API usage.
- This is direct-pattern focused. Keep it only when the diff shows the unsafe call, broken contract, or misuse pattern itself.

Reference examples:
- Reference example (`ComparisonWithCancelingSubExprExample.cpp`):
```
bool compare_xyz(unsigned short x, unsigned short y, unsigned short z) {
  return (x + y < x + z);  // x can be canceled
}
```
