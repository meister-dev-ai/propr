Review focus: User-controlled data may not be null terminated

What to look for:
- String operations on user-controlled strings can result in buffer overflow or buffer over-read.
- Built-in C string functions such as`strcpy`require that their input string arguments are null terminated. If the input string arguments are not null terminated, these functions will read/write beyond the length of the buffer containing the string, resulting in either buffer over-read or buffer overflow, respectively.
- Keep the check only when the changed code appears to introduce, remove, bypass, or misuse the risky behavior itself.
- Prefer explicit code evidence such as API calls, removed guards, changed data flow steps, changed arguments, or changed signatures over broad similarity.

Relevance threshold:
- Drop this check if the diff only changes naming, comments, formatting, or unrelated plumbing.
- Do not keep this check solely because the file belongs to a broad category like security, configuration, or API usage.
- This is direct-pattern focused. Keep it only when the diff shows the unsafe call, broken contract, or misuse pattern itself.

Reference examples:
- Reference example (`ImproperNullTerminationTaintedBad.cpp`):
```
char buf[BUF_SIZE];
read(fd, buf, BUF_SIZE);
char cpy[BUF_SIZE];
strcpy(cpy, buf);
```
- Reference example (`ImproperNullTerminationTaintedGood.cpp`):
```
char buf[BUF_SIZE];
int count = read(fd, buf, BUF_SIZE);
if (count >= 0) {
  buf[count < BUF_SIZE ? count : BUF_SIZE - 1] = '\0';
  char cpy[BUF_SIZE];
  strcpy(cpy, buf);
}
```
