Review focus: Comparison where assignment was intended

What to look for:
- The '==' operator may have been used accidentally, where '=' was intended, resulting in a useless test.
- This rule finds uses of the equality operator`==`in places where the assignment operator`=`would make more sense. This is a common mistake in C and C++, because of the similarity of the`=`and the`==`operator, and the fact that expressions are valid as top-level statements. The rule flags every occurrence of an equality operator in a position where its result is discarded.
- Keep the check only when the changed code appears to introduce, remove, bypass, or misuse the risky behavior itself.
- Prefer explicit code evidence such as API calls, removed guards, changed data flow steps, changed arguments, or changed signatures over broad similarity.

Relevance threshold:
- Drop this check if the diff only changes naming, comments, formatting, or unrelated plumbing.
- Do not keep this check solely because the file belongs to a broad category like security, configuration, or API usage.
- This is direct-pattern focused. Keep it only when the diff shows the unsafe call, broken contract, or misuse pattern itself.

Reference examples:
- Reference example (`CompareWhereAssignMeant.cpp`):
```
int x;
x == 4; // most likely = was intended. Otherwise this statement has no effect.
...
```
