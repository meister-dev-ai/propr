Review focus: Constant string literals

What to look for:
- A string literal must not be modified, as the result is undefined.
- This query highlights string literals that are assigned to a non-`const`variable. String literals should not be changed, since they are usually stored in the data section, and depending on the architecture, writing to the data section will cause undefined behavior, such as memory corruption or memory write error.
- Keep the check only when the changed code appears to introduce, remove, bypass, or misuse the risky behavior itself.
- Prefer explicit code evidence such as API calls, removed guards, changed data flow steps, changed arguments, or changed signatures over broad similarity.

Relevance threshold:
- Drop this check if the diff only changes naming, comments, formatting, or unrelated plumbing.
- Do not keep this check solely because the file belongs to a broad category like security, configuration, or API usage.
- This is direct-pattern focused. Keep it only when the diff shows the unsafe call, broken contract, or misuse pattern itself.

Reference examples:
- Reference example (`AV Rule 151.1.cpp`):
```
void f() {
	char *s = "String"; //wrong: literal assigned to non-const
	//this will cause a write error or corrupt other data in the data section
	strcpy(s, "Another string"); 

	const char* cs ="String"; //correct: literal assigned to a const
	//this will cause a compile error (trying to write to a const)
	strcpy(cs, "Another string");
}
```
