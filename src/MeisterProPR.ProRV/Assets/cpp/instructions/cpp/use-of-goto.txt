Review focus: Use of goto

What to look for:
- The goto statement can make the control flow of a function hard to understand, when used for purposes other than error handling.
- Use of`goto`statements makes code more difficult to understand and maintain. Consequently, the use of`goto`statements is deprecated except as a mechanism for breaking out of multiple nested loops, or jumping to error-handling code at the end of a function. This rule identifies complex (and therefore hard to understand) uses of`goto`statements. Function bodies that include multiple`goto`statements that jump forward and multiple`goto`statements that jump backwards are highlighted.
- Keep the check only when the changed code appears to introduce, remove, bypass, or misuse the risky behavior itself.
- Prefer explicit code evidence such as API calls, removed guards, changed data flow steps, changed arguments, or changed signatures over broad similarity.

Relevance threshold:
- Drop this check if the diff only changes naming, comments, formatting, or unrelated plumbing.
- Do not keep this check solely because the file belongs to a broad category like security, configuration, or API usage.
- This is direct-pattern focused. Keep it only when the diff shows the unsafe call, broken contract, or misuse pattern itself.
