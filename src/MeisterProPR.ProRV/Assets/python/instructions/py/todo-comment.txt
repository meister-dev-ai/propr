Review focus: 'To Do' comment

What to look for:
- Writing comments that include 'TODO' tends to lead to a build up of partially implemented features.
- A comment that includes the word`TODO`often marks a part of the code that is incomplete or broken, or highlights ambiguities in the software's specification. For example, this list of comments is typical of those found in real programs:
- -`TODO: move this code somewhere else`
- -`TODO: find a better solution to this workaround`
- -`TODO: test this`
- Keep the check only when the changed code appears to introduce, remove, bypass, or misuse the risky behavior itself.
- Prefer explicit code evidence such as API calls, removed guards, changed data flow steps, changed arguments, or changed signatures over broad similarity.

Relevance threshold:
- Drop this check if the diff only changes naming, comments, formatting, or unrelated plumbing.
- Do not keep this check solely because the file belongs to a broad category like security, configuration, or API usage.
- This is direct-pattern focused. Keep it only when the diff shows the unsafe call, broken contract, or misuse pattern itself.

Reference examples:
- Reference example (`ToDoComment.py`):
```
def realpath(path):
    '''
    Returns the true, canonical file system path equivalent to the given
    path.
    '''
    # TODO: There may be a more clever way to do this that also handles other,
    # less common file systems.
    return os.path.normpath(normcase(os.path.realpath(path)))
```
