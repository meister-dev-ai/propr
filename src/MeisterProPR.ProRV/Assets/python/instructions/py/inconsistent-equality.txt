Review focus: Inconsistent equality and inequality

What to look for:
- Class definitions of equality and inequality operators may be inconsistent.
- In order to ensure the`==`and`!=`operators behave consistently as expected (i.e. they should be negations of each other), care should be taken when implementing the`__eq__`and`__ne__`special methods. In Python 3, if the`__eq__`method is defined in a class while the`__ne__`is not, then the`!=`operator will automatically delegate to the`__eq__`method in the expected way. However, if the`__ne__`method is defined without a corresponding`__eq__`method, the`==`operator will still default to object identity (equivalent to the`is`operator), while the`!=`operator will use the`__ne__`method, which may be inconsistent. Additionally, if the`__ne__`method is defined on a superclass, and the subclass defines its own`__eq__`method without overriding the superclass`__ne__`method, the`!=`operator will use this superclass`__ne__`method, rather than automatically delegating to`__eq__`, which may be incorrect.
- Keep the check only when the changed code appears to introduce, remove, bypass, or misuse the risky behavior itself.
- Prefer explicit code evidence such as API calls, removed guards, changed data flow steps, changed arguments, or changed signatures over broad similarity.

Relevance threshold:
- Drop this check if the diff only changes naming, comments, formatting, or unrelated plumbing.
- Do not keep this check solely because the file belongs to a broad category like security, configuration, or API usage.
- This is direct-pattern focused. Keep it only when the diff shows the unsafe call, broken contract, or misuse pattern itself.

Reference examples:
- Reference example (`EqualsOrNotEquals1.py`):
```
class A:
    def __init__(self, a):
        self.a = a 

    # BAD: ne is defined, but not eq.
    def __ne__(self, other):
        if not isinstance(other, A):
            return NotImplemented 
        return self.a != other.a

x = A(1)
y = A(1)

print(x == y) # Prints False (potentially unexpected - object identity is used)
print(x != y) # Prints False
```
- Reference example (`EqualsOrNotEquals2.py`):
```
class B:
    def __init__(self, b):
        self.b = b 
    
    def __eq__(self, other):
        return self.b == other.b 
    
    def __ne__(self, other):
        return self.b != other.b 
    
class C(B):
    def __init__(self, b, c):
        super().__init__(b)
        self.c = c 

    # BAD: eq is defined, but != will use superclass ne method, which is not consistent
    def __eq__(self, other):
        return self.b == other.b and self.c == other.c 
    
print(C(1,2) == C(1,3)) # Prints False 
print(C(1,2) != C(1,3)) # Prints False (potentially unexpected)
```
