Review focus: Unsafe shell command constructed from library input

What to look for:
- Using externally controlled strings in a command line may allow a malicious user to change the meaning of the command.
- Dynamically constructing a shell command with inputs from library functions may inadvertently change the meaning of the shell command. Clients using the exported function may use inputs containing characters that the shell interprets in a special way, for instance quotes and spaces. This can result in the shell command misbehaving, or even allowing a malicious user to execute arbitrary commands on the system.
- Keep the check only when the changed code appears to introduce, remove, bypass, or misuse the risky behavior itself.
- Prefer explicit code evidence such as API calls, removed guards, changed data flow steps, changed arguments, or changed signatures over broad similarity.

Relevance threshold:
- Drop this check if the diff only changes naming, comments, formatting, or unrelated plumbing.
- Do not keep this check solely because the file belongs to a broad category like security, configuration, or API usage.
- This is path-sensitive. Keep it only when the diff suggests a plausible source-to-sink or input-to-operation path worth deeper inspection.

Reference examples:
- Reference example (`unsafe-shell-command-construction.py`):
```
import os

def download(path): 
    os.system("wget " + path) # NOT OK
```
- Reference example (`unsafe-shell-command-construction_fixed.py`):
```
import subprocess

def download(path): 
    subprocess.run(["wget", path]) # OK
```
