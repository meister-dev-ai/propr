Review focus: Clear-text logging of sensitive information

What to look for:
- Logging sensitive information without encryption or hashing can expose it to an attacker.
- If sensitive data is written to a log entry it could be exposed to an attacker who gains access to the logs. Potential attackers can obtain sensitive user data when the log output is displayed. Additionally that data may expose system information such as full path names, system information, and sometimes usernames and passwords.
- Keep the check only when the changed code appears to introduce, remove, bypass, or misuse the risky behavior itself.
- Prefer explicit code evidence such as API calls, removed guards, changed data flow steps, changed arguments, or changed signatures over broad similarity.

Relevance threshold:
- Drop this check if the diff only changes naming, comments, formatting, or unrelated plumbing.
- Do not keep this check solely because the file belongs to a broad category like security, configuration, or API usage.
- This is path-sensitive. Keep it only when the diff suggests a plausible source-to-sink or input-to-operation path worth deeper inspection.

Reference examples:
- Reference example (`CleartextLogging.py`):
```
# BAD: Logging cleartext sensitive data
import os
print(f"[INFO] Environment: {os.environ}")
```
- Reference example (`CleartextLoggingGood.py`):
```
not_sensitive_data = {'a': 1, 'b': 2}
# GOOD: it is fine to log data that is not sensitive
print(f"[INFO] Some object contains: {not_sensitive_data}")
```
