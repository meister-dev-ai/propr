Review focus: Unsafe expansion of self-closing HTML tag

What to look for:
- Using regular expressions to expand self-closing HTML tags may lead to cross-site scripting vulnerabilities.
- Sanitizing untrusted input for HTML meta-characters is a common technique for preventing cross-site scripting attacks. But even a sanitized input can be dangerous to use if it is modified further before a browser treats it as HTML. A seemingly innocent transformation that expands a self-closing HTML tag from``to``may in fact cause cross-site scripting vulnerabilities.
- Keep the check only when the changed code appears to introduce, remove, bypass, or misuse the risky behavior itself.
- Prefer explicit code evidence such as API calls, removed guards, changed data flow steps, changed arguments, or changed signatures over broad similarity.

Relevance threshold:
- Drop this check if the diff only changes naming, comments, formatting, or unrelated plumbing.
- Do not keep this check solely because the file belongs to a broad category like security, configuration, or API usage.
- This is direct-pattern focused. Keep it only when the diff shows the unsafe call, broken contract, or misuse pattern itself.

Reference examples:
- Reference example (`UnsafeHtmlExpansion.js`):
```
function expandSelfClosingTags(html) {
	var rxhtmlTag = /<(?!img|area)(([a-z][^\w\/>]*)[^>]*)\/>/gi;
	return html.replace(rxhtmlTag, "<$1></$2>"); // BAD
}
```
- Reference example (`UnsafeHtmlExpansion-original.html`):
```
<div alt="
<x" title="/>
<img src=url404 onerror=alert(1)>"/>
```
- Reference example (`UnsafeHtmlExpansion-transformed.html`):
