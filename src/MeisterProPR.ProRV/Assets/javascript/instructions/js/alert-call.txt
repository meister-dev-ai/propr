Review focus: Invocation of alert

What to look for:
- 'alert' should not be used in production code.
- Using the`alert`function to show a message to the user is disruptive, since an alert dialog prevents the user from accessing the rest of the program's interface until the dialog is closed. Also, alert dialogs are displayed by the browser and hence cannot be styled to fit with the rest of the web application, which negatively affects user experience. Some developers use`alert`during debugging, so left over calls to`alert`may also indicate unfinished code.
- Keep the check only when the changed code appears to introduce, remove, bypass, or misuse the risky behavior itself.
- Prefer explicit code evidence such as API calls, removed guards, changed data flow steps, changed arguments, or changed signatures over broad similarity.

Relevance threshold:
- Drop this check if the diff only changes naming, comments, formatting, or unrelated plumbing.
- Do not keep this check solely because the file belongs to a broad category like security, configuration, or API usage.
- This is direct-pattern focused. Keep it only when the diff shows the unsafe call, broken contract, or misuse pattern itself.

Reference examples:
- Reference example (`Alert.js`):
```
if (!/[\w.]+@[\w.]+\.\w+/.test(form.email.value))
	alert("Please enter a valid email address.");
```
