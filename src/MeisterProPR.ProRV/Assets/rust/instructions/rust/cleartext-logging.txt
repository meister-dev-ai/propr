Review focus: Cleartext logging of sensitive information

What to look for:
- Logging sensitive information in plaintext can expose it to an attacker.
- Sensitive user data and system information that is logged could be exposed to an attacker when it is displayed. Also, external processes often store the standard output and standard error streams of an application, which will include logged sensitive information.
- Keep the check only when the changed code appears to introduce, remove, bypass, or misuse the risky behavior itself.
- Prefer explicit code evidence such as API calls, removed guards, changed data flow steps, changed arguments, or changed signatures over broad similarity.

Relevance threshold:
- Drop this check if the diff only changes naming, comments, formatting, or unrelated plumbing.
- Do not keep this check solely because the file belongs to a broad category like security, configuration, or API usage.
- This is path-sensitive. Keep it only when the diff suggests a plausible source-to-sink or input-to-operation path worth deeper inspection.

Reference examples:
- Reference example (`CleartextLoggingBad.rs`):
```
let password = "P@ssw0rd";
info!("User password changed to {password}");
```
- Reference example (`CleartextLoggingGood.rs`):
```
let password = "P@ssw0rd";
info!("User password changed");
```
