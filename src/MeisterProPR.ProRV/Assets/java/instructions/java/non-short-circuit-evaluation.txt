Review focus: Dangerous non-short-circuit logic

What to look for:
- Using a bitwise logical operator on a boolean where a conditional-and or conditional-or operator is intended may yield the wrong result or cause an exception.
- Using a bitwise logical operator (`&`or`|`) on a boolean where a conditional-and or conditional-or operator (`&&`or`||`) is intended may yield the wrong result or cause an exception. This is especially true if the left-hand operand is a guard for the right-hand operand. Typically, as in the example below, this kind of defect is introduced by simply mistyping the intended logical operator rather than any conceptual mistake by the programmer.
- Keep the check only when the changed code appears to introduce, remove, bypass, or misuse the risky behavior itself.
- Prefer explicit code evidence such as API calls, removed guards, changed data flow steps, changed arguments, or changed signatures over broad similarity.

Relevance threshold:
- Drop this check if the diff only changes naming, comments, formatting, or unrelated plumbing.
- Do not keep this check solely because the file belongs to a broad category like security, configuration, or API usage.
- This is direct-pattern focused. Keep it only when the diff shows the unsafe call, broken contract, or misuse pattern itself.
