Review focus: Auto boxing or unboxing

What to look for:
- Implicit boxing or unboxing of primitive types, such as 'int' and 'double', may cause confusion and subtle performance problems.
- For each primitive type, such as`int`or`double`, there is a corresponding boxed reference type, such as`Integer`or`Double`. These boxed versions differ from their primitive equivalents because they can hold an undefined`null`element in addition to numeric (or other) values, and there can be more than one instance of a boxed type representing the same value. In Java 5 and later, automated boxing and unboxing conversions have been added to the language. Although these automated conversions reduce the verbosity of the code, they can hide potential problems. Such problems include performance issues because of unnecessary object creation, and confusion of boxed types with their primitive equivalents.
- Keep the check only when the changed code appears to introduce, remove, bypass, or misuse the risky behavior itself.
- Prefer explicit code evidence such as API calls, removed guards, changed data flow steps, changed arguments, or changed signatures over broad similarity.

Relevance threshold:
- Drop this check if the diff only changes naming, comments, formatting, or unrelated plumbing.
- Do not keep this check solely because the file belongs to a broad category like security, configuration, or API usage.
- This is direct-pattern focused. Keep it only when the diff shows the unsafe call, broken contract, or misuse pattern itself.
