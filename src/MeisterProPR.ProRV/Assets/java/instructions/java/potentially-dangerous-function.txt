Review focus: Use of a potentially dangerous function

What to look for:
- Certain standard library routines are dangerous to call.
- This rule finds calls to methods that are dangerous to use. Currently, it checks for calls to`Thread.stop`. Stopping a thread with`Thread.stop`causes it to receive a`ThreadDeath`exception. That exception propagates up the stack, releasing all monitors that the thread was holding. In some cases the relevant code will be protected by catching the`ThreadDeath`exception and cleaning up, but because the exception can potentially be thrown from so very many locations, it is impractical to catch all such cases. As a result, calling`Thread.stop`is likely to result in corrupt data.
- Keep the check only when the changed code appears to introduce, remove, bypass, or misuse the risky behavior itself.
- Prefer explicit code evidence such as API calls, removed guards, changed data flow steps, changed arguments, or changed signatures over broad similarity.

Relevance threshold:
- Drop this check if the diff only changes naming, comments, formatting, or unrelated plumbing.
- Do not keep this check solely because the file belongs to a broad category like security, configuration, or API usage.
- This is direct-pattern focused. Keep it only when the diff shows the unsafe call, broken contract, or misuse pattern itself.
