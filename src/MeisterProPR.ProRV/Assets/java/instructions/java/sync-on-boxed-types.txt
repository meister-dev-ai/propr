Review focus: Synchronization on boxed types or strings

What to look for:
- Synchronizing on boxed types or strings may lead to deadlock since an instance of that type is likely to be shared between many parts of the program.
- Code should not synchronize on a variable or field of a boxed type (for example`Integer`,`Boolean`) or of type`String`since it is likely to contain an object that is used throughout the program. For example,`Boolean.TRUE`holds a single instance that will be used in many places throughout the program: whenever`true`is autoboxed or a call to`Boolean.valueOf`is made with`true`as an argument the same instance of`Boolean`is returned. It is therefore likely that two classes synchronizing on a field of type`Boolean`will end up synchronizing on the same object. This may lead to deadlock or threads being blocked unnecessarily.
- Keep the check only when the changed code appears to introduce, remove, bypass, or misuse the risky behavior itself.
- Prefer explicit code evidence such as API calls, removed guards, changed data flow steps, changed arguments, or changed signatures over broad similarity.

Relevance threshold:
- Drop this check if the diff only changes naming, comments, formatting, or unrelated plumbing.
- Do not keep this check solely because the file belongs to a broad category like security, configuration, or API usage.
- This is direct-pattern focused. Keep it only when the diff shows the unsafe call, broken contract, or misuse pattern itself.
