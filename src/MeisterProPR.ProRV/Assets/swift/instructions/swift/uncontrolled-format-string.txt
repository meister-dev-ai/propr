Review focus: Uncontrolled format string

What to look for:
- Using external input in format strings can lead to exceptions or information leaks.
- Passing untrusted format strings to functions that use`printf`style formatting can lead to buffer overflows and data representation problems. An attacker may be able to exploit this weakness to crash the program or obtain sensitive information from its internal state.
- Keep the check only when the changed code appears to introduce, remove, bypass, or misuse the risky behavior itself.
- Prefer explicit code evidence such as API calls, removed guards, changed data flow steps, changed arguments, or changed signatures over broad similarity.

Relevance threshold:
- Drop this check if the diff only changes naming, comments, formatting, or unrelated plumbing.
- Do not keep this check solely because the file belongs to a broad category like security, configuration, or API usage.
- This is path-sensitive. Keep it only when the diff suggests a plausible source-to-sink or input-to-operation path worth deeper inspection.

Reference examples:
- Bad example (`UncontrolledFormatStringBad.swift`):
```
print(String(format: "User input: " + inputString)) // vulnerable
```
- Good example (`UncontrolledFormatStringGood.swift`):
```
print(String(format: "User input: %@", inputString)) // fixed
```
