Review focus: Unsafe WebView fetch

What to look for:
- Fetching data in a WebView without restricting the base URL may allow an attacker to access sensitive local data, or enable cross-site scripting attack.
- Fetching data in a web view without restricting the base URL may allow an attacker to access sensitive local data, for example using`file://`. Data can then be extracted from the software using the URL of a machine under the attacker's control. More generally, an attacker may use a URL under their control as part of a cross-site scripting attack.
- Keep the check only when the changed code appears to introduce, remove, bypass, or misuse the risky behavior itself.
- Prefer explicit code evidence such as API calls, removed guards, changed data flow steps, changed arguments, or changed signatures over broad similarity.

Relevance threshold:
- Drop this check if the diff only changes naming, comments, formatting, or unrelated plumbing.
- Do not keep this check solely because the file belongs to a broad category like security, configuration, or API usage.
- This is path-sensitive. Keep it only when the diff suggests a plausible source-to-sink or input-to-operation path worth deeper inspection.

Reference examples:
- Bad example (`UnsafeWebViewFetchBad.swift`):
```
let webview = UIWebView()

...

webview.loadHTMLString(htmlData, baseURL: nil) // BAD
```
- Good example (`UnsafeWebViewFetchGood.swift`):
```
let webview = UIWebView()

...

webview.loadHTMLString(htmlData, baseURL: URL(string: "about:blank")) // GOOD
```
