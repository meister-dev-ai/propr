Review focus: Missed ternary opportunity

What to look for:
- An 'if' statement where both branches either (a) return or (b) write to the same variable can often be expressed more clearly using the '?' operator.
- An`if`statement where both branches do nothing but return or write to a variable can be difficult to read. It takes up a lot of lines of code and, in the case of assignment, requires the declaration of the variable to be on a different line to the two assignments. It also does not adequately express the intent of the programmer to assign or return a value based on a condition.
- Keep the check only when the changed code appears to introduce, remove, bypass, or misuse the risky behavior itself.
- Prefer explicit code evidence such as API calls, removed guards, changed data flow steps, changed arguments, or changed signatures over broad similarity.

Relevance threshold:
- Drop this check if the diff only changes naming, comments, formatting, or unrelated plumbing.
- Do not keep this check solely because the file belongs to a broad category like security, configuration, or API usage.
- This is direct-pattern focused. Keep it only when the diff shows the unsafe call, broken contract, or misuse pattern itself.

Reference examples:
- Reference example (`MissedTernaryOpportunity.cs`):
```
class MissedTernaryOpportunity
{
    static int MyAbs(int x)
    {
        if (x >= 0)
            return x;
        else
            return -x;
    }
}
```
- Reference example (`MissedTernaryOpportunityFix.cs`):
```
class MissedTernaryOpportunityFix
{
    static int MyAbs(int x)
    {
        return x >= 0 ? x : -x;
    }
}
```
