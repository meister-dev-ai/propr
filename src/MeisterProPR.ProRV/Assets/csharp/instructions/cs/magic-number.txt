Review focus: Magic numbers

What to look for:
- 'Magic constants' should be avoided: if a nontrivial constant is used repeatedly, it should be encapsulated into a const variable or macro definition.
- This rule finds numbers that are used multiple times that should possibly be constants instead. Named constants make it easier to determine the purpose of the number and make the number easy to change if it is used more than once.
- Keep the check only when the changed code appears to introduce, remove, bypass, or misuse the risky behavior itself.
- Prefer explicit code evidence such as API calls, removed guards, changed data flow steps, changed arguments, or changed signatures over broad similarity.

Relevance threshold:
- Drop this check if the diff only changes naming, comments, formatting, or unrelated plumbing.
- Do not keep this check solely because the file belongs to a broad category like security, configuration, or API usage.
- This is direct-pattern focused. Keep it only when the diff shows the unsafe call, broken contract, or misuse pattern itself.

Reference examples:
- Reference example (`MagicConstantsNumbers.cs`):
```
class Circle
{
    private double radius;
    public double area()
    {
        return Math.Pow(radius, 2) * 3.14;
    }
    public double circumference()
    {
        return radius * 2 * 3.14;
    }
}
```
