Review focus: Value shadowing: server variable

What to look for:
- Finds ambiguous accesses to server variables
- Relying on`HttpRequest`to provide access to a particular server variable is not safe as it can be overridden by the client. The`HttpRequest` class implements an indexer to provide a simplified, combined access to its`QueryString`,`Form`,`Cookies`, or`ServerVariables`collections, in that particular order. When searching for a variable, the first match is returned:`QueryString`parameters hence supersede values from forms, cookies and server variables, and so on. This is a serious attack vector since an attacker could inject a value in the query string that you do not expect, and which supersedes the value of the server variable you were actually trying to check.
- Keep the check only when the changed code appears to introduce, remove, bypass, or misuse the risky behavior itself.
- Prefer explicit code evidence such as API calls, removed guards, changed data flow steps, changed arguments, or changed signatures over broad similarity.

Relevance threshold:
- Drop this check if the diff only changes naming, comments, formatting, or unrelated plumbing.
- Do not keep this check solely because the file belongs to a broad category like security, configuration, or API usage.
- This is direct-pattern focused. Keep it only when the diff shows the unsafe call, broken contract, or misuse pattern itself.

Reference examples:
- Reference example (`ValueShadowingServerVariable.cs`):
```
class ValueShadowingServerVariable
{
    public bool isHTTPS(HttpRequest request)
    {
        String https = request["HTTPS"];
        return https == "ON";
    }
}
```
- Reference example (`ValueShadowingServerVariableFix.cs`):
```
class ValueShadowingServerVariableFix
{
    public bool isHTTPS(HttpRequest request)
    {
        String https = request.ServerVariables["HTTPS"];
        return https == "ON";
    }
}
```
