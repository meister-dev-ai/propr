Review focus: Inconsistent Equals(object) and GetHashCode()

What to look for:
- If a class overrides only one of 'Equals(object)' and 'GetHashCode()', it may mean that 'Equals(object)' and 'GetHashCode()' are inconsistent.
- A class that overrides only one of`Equals(object)`and`GetHashCode()`is likely to violate the contract of the`GetHashCode()`method. The contract requires that`GetHashCode()`gives the same integer result for any two equal objects. Not enforcing this property may cause unexpected results when storing and retrieving objects of such a class in a hashing data structure.
- Keep the check only when the changed code appears to introduce, remove, bypass, or misuse the risky behavior itself.
- Prefer explicit code evidence such as API calls, removed guards, changed data flow steps, changed arguments, or changed signatures over broad similarity.

Relevance threshold:
- Drop this check if the diff only changes naming, comments, formatting, or unrelated plumbing.
- Do not keep this check solely because the file belongs to a broad category like security, configuration, or API usage.
- This is direct-pattern focused. Keep it only when the diff shows the unsafe call, broken contract, or misuse pattern itself.

Reference examples:
- Reference example (`InconsistentEqualsGetHashCodeBad.cs`):
```
using System;

class Bad
{
    private int id;

    public Bad(int Id) { this.id = Id; }

    public override bool Equals(object other)
    {
        if (other is Bad b && b.GetType() == typeof(Bad))
            return this.id == b.id;
        return false;
    }
}
```
- Reference example (`InconsistentEqualsGetHashCodeGood.cs`):

using System;

class Good
