Review focus: Incorrect type parameter name in documentation

What to look for:
- The type parameter name given in a `<typeparam>` tag does not exist.
- Documentation comments for public classes and methods should use the``tag to describe the available type parameters. If the comment includes any empty, incorrect or outdated parameter names then this will make the documentation more difficult to read.
- Keep the check only when the changed code appears to introduce, remove, bypass, or misuse the risky behavior itself.
- Prefer explicit code evidence such as API calls, removed guards, changed data flow steps, changed arguments, or changed signatures over broad similarity.

Relevance threshold:
- Drop this check if the diff only changes naming, comments, formatting, or unrelated plumbing.
- Do not keep this check solely because the file belongs to a broad category like security, configuration, or API usage.
- This is direct-pattern focused. Keep it only when the diff shows the unsafe call, broken contract, or misuse pattern itself.

Reference examples:
- Reference example (`XmldocTypeParam.cs`):
```
/// <summary>
///   BAD: An example of a class that lacks documentation
///   for a type parameter.
/// </summary>
class Stack<T>
{
}

/// <summary>
///   BAD: An example of a class that has incorrect documentation
///   for the type parameter.
/// </summary>
/// <typeparam name="X">The type of each stack element.</typeparam>
class Stack<T>
{
}

/// <summary>
///   GOOD: An example of a class whose type parameters are
///   correctly documented.
/// </summary>
/// <typeparam name="T">The type of each stack element.</typeparam>
class Stack<T>
{
}
```
