Review focus: Call to ReferenceEquals(...) on value type expressions

What to look for:
- 'ReferenceEquals(...)' always returns false on value types - its use is at best redundant and at worst erroneous.
- The`Object.ReferenceEquals(...)`method is used to determine if two objects are the same instance. Since the method takes two objects as parameters, value types will automatically be boxed in different objects when calling the method. Hence, the method`ReferenceEquals(..)`will always return`false`when comparing value type expressions, including struct values. As such, this kind of check is at best useless and at worst erroneous.
- Keep the check only when the changed code appears to introduce, remove, bypass, or misuse the risky behavior itself.
- Prefer explicit code evidence such as API calls, removed guards, changed data flow steps, changed arguments, or changed signatures over broad similarity.

Relevance threshold:
- Drop this check if the diff only changes naming, comments, formatting, or unrelated plumbing.
- Do not keep this check solely because the file belongs to a broad category like security, configuration, or API usage.
- This is direct-pattern focused. Keep it only when the diff shows the unsafe call, broken contract, or misuse pattern itself.
