Review focus: Null argument to Equals(object)

What to look for:
- Calls of the form 'o.Equals(null)' always return false for non-null 'o', and throw a 'NullReferenceException' when 'o' is null.
- It is common to want to check an object against`null`, but this should not be done using the`Equals`method. If the object really is`null`, a`NullReferenceException`is thrown when attempting to call`Equals`, with unexpected results.
- Keep the check only when the changed code appears to introduce, remove, bypass, or misuse the risky behavior itself.
- Prefer explicit code evidence such as API calls, removed guards, changed data flow steps, changed arguments, or changed signatures over broad similarity.

Relevance threshold:
- Drop this check if the diff only changes naming, comments, formatting, or unrelated plumbing.
- Do not keep this check solely because the file belongs to a broad category like security, configuration, or API usage.
- This is direct-pattern focused. Keep it only when the diff shows the unsafe call, broken contract, or misuse pattern itself.

Reference examples:
- Reference example (`NullArgumentToEqualsBad.cs`):
```
class Bad
{
    bool IsNull(object o) => o.Equals(null);
}
```
- Reference example (`NullArgumentToEqualsGood.cs`):
```
class Good
{
    bool IsNull(object o) => o == null;
}
```
