# Testing Patterns

## Test Frameworks

- Backend tests use xUnit with NSubstitute.
- Frontend unit/component tests use Vitest 3 with `@vue/test-utils`.
- Browser-level frontend flows use Playwright under `frontend/tests/e2e/`.

## API Host Tests

- API integration tests usually boot `WebApplicationFactory<Program>` in the `Testing` environment with `MEISTER_DISABLE_HOSTED_SERVICES=true` plus fake AI settings.
- Most full-host API tests authenticate with bearer tokens generated by the test fixture (`GenerateAdminToken()`, tenant/client user tokens, and similar helpers). Do not assume auth is always bypassed.
- Direct controller or narrow unit tests still sometimes set `HttpContext.Items["IsAdmin"]`, `HttpContext.Items["UserId"]`, `HttpContext.Items["ClientRoles"]`, or request headers manually when they are testing controller behavior without the full middleware pipeline. That is also intentional.
- `MeisterProPR.ProCursor.Service.Tests` boot the extracted ProCursor host separately. Do not force ProCursor-host behavior through the API host when a dedicated service-host test already fits better.

## Database Patterns

- Many API and slice tests override DI to use `UseInMemoryDatabase(...)`, often with a shared `InMemoryDatabaseRoot` per factory so state persists across multiple requests in one test flow.
- DB-backed tests use PostgreSQL via Testcontainers (`pgvector/pgvector:pg17`) or an externally supplied `DB_CONNECTION_STRING`. Those fixtures run real migrations and may skip when Docker is unavailable.
- The repo intentionally mixes in-memory and real-PostgreSQL coverage. Use in-memory for fast host wiring/controller scenarios, and PostgreSQL when migrations, JSONB, SQL/provider behavior, or persistence semantics matter.
- Because DB-backed registrations are enabled whenever `DB_CONNECTION_STRING` is present, tests that expect in-memory wiring should avoid inheriting a real connection string or explicitly override the affected registrations.

## Secret And Auth Assertions

- Write-only secrets must stay absent from API GET responses.
- Data Protection-backed secrets should remain protected at rest where the slice stores encrypted secrets.
- PATs and refresh tokens follow different rules: PATs are BCrypt-hashed and refresh tokens are stored by SHA-256 hash rather than reversible encryption.
- Startup and security tests in this repo often validate migrations, backfill behavior, JWT auth, PAT auth, or secret-protection behavior instead of only asserting controller status codes.

## NSubstitute Conventions

- `Substitute.For<T>()`, `Arg.Any<T>()`, `Arg.Is<T>(...)`, `Received()`, `DidNotReceive()`, and `DidNotReceiveWithAnyArgs()` are all standard in this codebase.
- `DidNotReceiveWithAnyArgs()` followed by placeholder `null!` arguments is intentional NSubstitute syntax, not a null-handling bug.
- `Returns(...)` is the specific setup; `ReturnsForAnyArgs(...)` is the broader fallback.

## Common Backend Test Patterns

- Shared `InMemoryDatabaseRoot` instances are intentional when a factory needs persistence across multiple requests.
- Test factories commonly replace external/provider dependencies with substitutes while leaving the rest of the host graph real.
- `MEISTER_DISABLE_HOSTED_SERVICES=true` is the normal way to prevent background workers from interfering with host-based tests.

## Frontend Tests

- Frontend tests commonly mock service modules, router helpers, and `useSession()` with `vi.mock()` rather than booting the whole app.
- MSW handlers and browser-storage-backed mock state are intentional parts of local/test flows.
- `flushPromises()` resolves pending promise chains but does not advance timers. For timer-driven behavior, also use `vi.useFakeTimers()` and `vi.advanceTimersByTimeAsync()`.
